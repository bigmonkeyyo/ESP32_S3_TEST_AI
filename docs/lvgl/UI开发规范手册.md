# UI开发规范手册（LVGL / ESP32-S3）

## 1. 目标与适用范围
- 目标：提高页面开发一致性，减少生命周期/动画/线程相关崩溃，降低后续智能体接手成本。
- 适用范围：`components/UI/*`、`components/Middlewares/lvgl_port/*`、`components/BSP/LCD/*`。

## 2. 架构边界
- 页面文件（`components/UI/pages`）只做页面元素创建与业务交互，不直接操作底层显示驱动。
- 页面切换统一走 `ui_page_manager` 接口：
- `ui_page_push(...)`
- `ui_page_pop(...)`
- `ui_page_replace(...)`
- `ui_page_back_to_root(...)`
- 禁止在页面回调中直接调用 `lv_scr_load*` 或直接管理页面栈。

## 3. 页面生命周期规范
- 页面描述符必须完整定义：
- `create`
- `cache_mode`
- `on_show/on_hide`（按需）
- `on_destroy`（至少将静态页面指针置空）
- 页面对象销毁规则：
- 非缓存页（`UI_PAGE_CACHE_NONE`）允许销毁。
- 缓存页（`UI_PAGE_CACHE_KEEP`）禁止销毁。
- 动画场景下，旧页面销毁必须延后到动画完成窗口外，禁止“立即删”。
- 当前项目默认使用 `lv_obj_del_delayed(..., 260ms)`，应保持 `删除延迟 >= 页面切换动画时长 + 安全裕量`。

## 4. 事件回调规范
- 事件回调只做轻量动作（切页、状态切换、发消息）。
- 禁止在点击回调内做以下高风险动作：
- 同步删除当前正在参与事件分发/布局的对象树
- 长时间阻塞（如耗时IO、死循环、长延时）
- 页面返回按钮的标准动作：调用 `ui_page_pop(UI_ANIM_MOVE_RIGHT)`，不得自定义删除逻辑。

## 5. 动画与删除协同规范
- `ui_page_anim_load` 触发切页动画后，旧页面仍可能被 LVGL 布局或事件路径引用。
- 因此必须遵守：
- 切页后旧页删除采用“异步或延后”策略。
- 优先使用“延后删除”，避免“下一 tick 删除”与布局刷新冲突。
- 若修改动画时长（当前约 220ms），必须同步评估删除延时常量。

## 6. 线程与调用上下文
- LVGL API 统一在 LVGL 任务上下文调用。
- 中断或其他线程如需触发 UI 行为，需转发到 LVGL 线程，不可直接跨线程操作 LVGL 对象。

## 7. 性能与显示规范
- 刷新完成信号必须以实际传输完成为准：
- `lv_disp_flush_ready()` 由 DMA/总线传输完成回调触发。
- 首帧点亮策略：
- 上电先关背光
- 首帧完成后开背光
- 需保留超时兜底，防止黑屏等待
- 显示总线频率调整（如 10MHz -> 20MHz）必须在目标板上执行稳定性回归（至少冷启动、切页、连续点击、长时间运行）。

## 8. 提交前检查清单
- 页面切换路径是否全部使用 `ui_page_manager`。
- 是否存在页面回调内直接删除对象。
- `cache_mode` 与销毁策略是否一致。
- 动画时长与删除延时是否匹配。
- 是否跑过最小回归：
- 冷启动 10 次
- `Home -> Settings/About -> Back` 连续 100 次
- 10 分钟持续操作无复位

## 9. 本项目已验证关键约束
- 返回卡死类问题：通过“旧页延后删除”规避。
- 白屏问题：通过“首帧后开背光 + 超时兜底”规避。
- 刷新卡顿问题：通过“flush_ready 绑定传输完成回调 + 双缓冲 + 总线调优”改善。
